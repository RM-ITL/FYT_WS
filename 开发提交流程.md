# 开发提交流程

这份文档是给当前仓库 `/home/hou/FYT_WS` 用的日常开发提交流程。

目标：
- 让每次在 VS Code 里修改代码之后，都能留下清晰的本地和 GitHub 迭代记录。
- 避免把临时文件、构建产物、无关改动一起提交。

```bash
cd /home/hou/FYT_WS
```

```bash
git@github.com:RM-ITL/FYT_WS.git
```

当前默认分支：

## 二、最简单的日常流程

每次改完一轮代码，执行：

```bash
cd /home/hou/FYT_WS
git status
git add .
git commit -m "feat: 这次修改的内容"
git push
```

这 4 步就足够保存你的迭代记录。

---

## 三、推荐使用的标准流程

建议每次开发按下面顺序来。

### 1. 开始改代码前先同步远端

```bash
cd /home/hou/FYT_WS
git pull --rebase
```

作用：
- 避免你本地改完之后，推送时发现远端先更新了。
- 保持提交历史更干净。

### 2. 修改代码

可以直接在 VS Code 里改。

### 3. 修改完成后先检查改动

```bash
git status
git diff
```

作用：
- 看哪些文件被改了。
- 检查有没有误改文件。
- 避免把无关内容提交进去。

### 4. 提交本次迭代

```bash
git add .
git commit -m "fix: 修正串口姿态时间戳对齐"
```

### 5. 推送到 GitHub

```bash
git push
```

这样这一次迭代就完整保存下来了。

---

## 四、提交信息怎么写

建议统一用：

```bash
类型: 这次修改做了什么
```

常用类型：

- `feat`: 新功能
- `fix`: 修 bug
- `refactor`: 重构
- `docs`: 文档修改
- `chore`: 杂项维护
- `test`: 测试相关

例子：

```bash
git commit -m "feat: 新增YOLO装甲板检测前端"
git commit -m "fix: 修正armor_solver弹速补偿逻辑"
git commit -m "refactor: 重构armor_tracker状态机"
git commit -m "docs: 更新FYT_WS改造文档"
git commit -m "chore: 调整gitignore忽略规则"
```

提交信息要求：
- 简短
- 具体
- 一眼能看懂这次干了什么

不要写这种低信息内容：

```bash
git commit -m "update"
git commit -m "修改代码"
git commit -m "test"
```

## 五、什么时候提交一次

建议按“一个完整的小改动”提交一次。

适合单独提交的粒度：
- 修完一个 bug
- 完成一个小功能
- 调完一轮可复现的参数
- 重构完一个模块
- 改完一批同类型文件

不建议：
- 攒很多天才提交一次
- 很多完全无关的改动混成一个提交

推荐习惯：
- 能跑通一轮就提交
- 能说明白一件事就提交

## 六、VS Code 里怎么操作

如果你不用命令行，也可以直接在 VS Code 里完成。

步骤：

1. 打开左侧 `Source Control`
2. 查看修改文件
3. 点开差异确认改动内容
4. 在输入框里写提交信息
5. 点击 `Commit`
6. 点击 `Sync Changes` 或 `Push`

建议：
- 提交前仍然看一眼 diff
- 不要盲目点全部提交

---

## 七、如果只想提交部分文件

如果这次只想提交部分改动，不要直接 `git add .`。

可以用：

```bash
git add 文件路径
git commit -m "fix: 只提交这一部分"
```

例如：

```bash
git add src/rm_auto_aim/armor_solver/src/armor_solver.cpp
git add src/rm_bringup/config/node_params/armor_solver_params.yaml
git commit -m "fix: 调整armor_solver延迟补偿参数"
```

---

## 八、如果提交之后又发现还要补一点

直接继续改，然后再提一个新提交：

```bash
git add .
git commit -m "fix: 补充边界条件处理"
git push
```

不建议现在频繁改历史、改上一个提交。

你的目标是保留清晰迭代记录，不是把历史修饰得非常漂亮。

---

## 九、如果 push 失败怎么办

### 1. 远端比你本地新

先执行：

```bash
git pull --rebase
git push
```

### 2. 有冲突

先看冲突文件：

```bash
git status
```

解决完冲突后：

```bash
git add .
git rebase --continue
git push
```

### 3. 不确定怎么处理

先不要乱执行 `reset --hard`。

先看：

```bash
git status
git diff
```

然后再处理。

## 十、不建议做的事

不要轻易执行这些命令：

```bash
git reset --hard
git push --force
git checkout -- 文件
```

除非你非常明确知道自己在做什么。

你现在已经把仓库推到了 GitHub，后续目标是“稳定留痕”，不是“频繁重写历史”。

## 十二、推荐你长期保持的习惯

- 每次提交前先看 `git diff`
- 每次提交信息写清楚
- 每天至少 push 一次
- 每完成一个小功能就 commit
- 配置、代码、文档如果是同一件事，可以放一个提交
- 无关改动不要混在同一个提交里

查看状态：

```bash
git status
```

查看改动：

```bash

git add .
```
